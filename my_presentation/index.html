IVF AI Chatbot

Healthcare Domain | RAG-Based Solution

üéØ Problem Statement

IVF patients and healthcare staff face significant challenges in accessing timely, accurate information about complex fertility treatments. Traditional support systems suffer from:

  • Long wait times for medical query responses, causing patient anxiety
  • Inconsistent information across different communication channels
  • Staff burnout from handling repetitive questions 24/7
  • Risk of misinformation when patients search unverified online sources
  • Limited availability of expert consultation outside clinic hours

üí° Solution Overview: RAG Architecture

Our Retrieval-Augmented Generation (RAG) based AI chatbot provides instant, accurate responses to IVF-related medical questions. The system works by:

  • Document Retrieval: Searching through verified IVF medical documents and protocols stored in a vector database
  • Context Extraction: Retrieving only the most relevant information chunks related to the patient's query
  • Grounded Generation: Using a Large Language Model (LLM) to generate natural language answers strictly based on retrieved documents
  • Citation Support: Every response includes source references for transparency and trust

This approach ensures responses are factually grounded in verified medical literature rather than generated from the LLM's training data alone.

üõ°Ô∏è Healthcare Safety & Hallucination Control

Patient safety is our highest priority. The system implements multiple safeguards to prevent AI hallucination:

  • Temperature Setting: LLM configured with temperature=0.0 for deterministic, consistent outputs
  • Document-Grounded Responses: Answers generated only from retrieved verified IVF documents, not from general knowledge
  • Confidence Thresholds: System explicitly states "I don't have information about this" when relevant documents aren't found
  • Source Attribution: Every answer traces back to specific medical documents for accountability
  • Medical Review Process: Content verified by IVF specialists before being added to the knowledge base
  • No Diagnostic Claims: System provides information but explicitly directs patients to consult their healthcare provider for medical decisions

High Level Design (HLD)

System Architecture Overview

Core Components

  • User Interface: Web-based chat interface where patients and staff interact with the chatbot through natural language queries
  • Embedding Model: Converts both user queries and document chunks into 384-dimensional dense vectors for semantic similarity matching
  • Vector Database (FAISS): Local vector store that enables fast similarity search across thousands of document embeddings to find relevant context
  • Large Language Model (LLM): GPT-3.5/GPT-4 model that generates coherent, contextual responses based on retrieved documents and user queries

Architecture Flow

When a user submits a question, the query flows through the system as follows: The embedding model converts the question into a vector representation. This vector is used to search the FAISS vector database for the most semantically similar document chunks. The top relevant chunks (typically 3-5) are retrieved and combined with the user's original question. This enriched context is passed to the LLM, which generates a natural language response grounded in the retrieved documents. The response is then displayed to the user through the interface, along with source citations.

Low Level Design (LLD)

Detailed Internal Flow

Document Ingestion Pipeline

Step 1: PDF Ingestion - Medical PDF documents containing IVF protocols, treatment guidelines, and FAQs are loaded using PyPDF or similar libraries. The system extracts raw text while preserving document structure and metadata.

Step 2: Text Chunking - Extracted text is split into smaller, semantically meaningful chunks (typically 500-1000 characters with 100-character overlap). This chunking strategy ensures that each piece of information is self-contained while maintaining context across chunk boundaries.

Step 3: Embedding Generation - Each text chunk is processed through the sentence-transformers model (all-MiniLM-L6-v2) to generate a 384-dimensional embedding vector. These embeddings capture the semantic meaning of the text, allowing for similarity-based retrieval.

Step 4: Vector Storage - All chunk embeddings along with their original text and metadata are stored in the FAISS vector database. FAISS creates an index structure optimized for fast similarity search at query time.

Query Processing & Response Generation

Step 1: Query Embedding - User's question is converted into a 384-dimensional vector using the same embedding model used during document ingestion, ensuring vector space compatibility.

Step 2: Vector Similarity Search - FAISS performs cosine similarity search between the query embedding and all stored document embeddings. The top-k most similar chunks (k=3 to 5) are retrieved based on similarity scores.

Step 3: Context Preparation - Retrieved chunks are combined with the original user query into a structured prompt template. This prompt includes instructions for the LLM to answer based only on provided context.

Step 4: LLM Answer Generation - The complete prompt is sent to GPT-3.5/GPT-4 with temperature=0.0 for deterministic output. The LLM synthesizes the retrieved information into a coherent, natural language response.

Step 5: Response Formatting - The generated answer is formatted with source citations, displaying which document chunks were used. If no relevant documents were found, the system responds with "I don't have information about this topic in my knowledge base."

Technology Stack

Technical Decisions and Justifications

Programming Language

Python 3.9+

Python was chosen for its rich ecosystem of NLP and machine learning libraries. It offers seamless integration with LangChain, transformers, and FAISS, while providing excellent development speed and readability for rapid prototyping and maintenance.

Embedding Model

sentence-transformers/all-MiniLM-L6-v2

This model generates high-quality 384-dimensional embeddings with excellent semantic accuracy. It's lightweight (only 80MB), runs efficiently on CPU, and provides a strong balance between performance and resource consumption, making it ideal for local deployment.

Vector Store

FAISS (Facebook AI Similarity Search)

FAISS enables extremely fast similarity search across millions of vectors with minimal latency. It runs locally without external dependencies, supports multiple index types for optimization, and requires no cloud connectivity, ensuring data privacy for sensitive medical information.

Large Language Model

GPT-3.5-turbo / GPT-4

OpenAI's models provide state-of-the-art natural language generation capabilities. Running with temperature=0.0 ensures deterministic, consistent outputs. These models excel at understanding medical context and generating coherent responses while following strict prompting instructions to avoid hallucination.

Framework

LangChain

LangChain simplifies RAG pipeline orchestration by providing pre-built components for document loading, text splitting, embeddings, vector stores, and LLM chains. It reduces boilerplate code, handles prompt templating elegantly, and offers robust error handling for production deployments.

Presentation Layer

HTML + CSS + JavaScript

Pure web technologies ensure maximum compatibility across all browsers and devices. This approach eliminates framework dependencies, reduces bundle size, and provides full control over the user experience. The single-page design enables easy deployment and sharing of the project presentation.

üìö Technical Explanation

Deep Dive: What I Built and Why

Technical Decisions and Reasoning

This video provides a comprehensive explanation of the architectural choices and implementation details:

  • Why RAG over Fine-Tuning: RAG architecture allows for dynamic knowledge updates without retraining, ensures responses are grounded in verifiable sources, and prevents hallucination by constraining the model to retrieved context. Fine-tuning would be expensive, time-consuming, and still risk hallucination.
  • Why FAISS over Cloud Vector Databases: Local deployment ensures patient data privacy compliance with HIPAA regulations, eliminates API latency and costs, and provides full control over the infrastructure. FAISS offers production-grade performance with minimal overhead.
  • Why sentence-transformers/all-MiniLM-L6-v2: This model provides an optimal balance between accuracy and efficiency. It generates high-quality embeddings while being lightweight enough to run on CPU, reducing infrastructure costs. The 384-dimensional vectors are sufficient for medical domain similarity matching.
  • Why Temperature=0.0: Healthcare applications demand deterministic, reproducible outputs. Temperature=0.0 eliminates randomness in the LLM's generation process, ensuring consistent answers for the same query and context, which is critical for patient safety.
  • Why LangChain Framework: LangChain provides battle-tested abstractions for RAG pipelines, reducing development time by 60-70%. It offers robust error handling, seamless integration with multiple LLMs and vector stores, and excellent documentation for maintenance.
  • Chunk Size Strategy (500-1000 chars): This range ensures each chunk contains complete semantic units while maintaining sufficient context. Smaller chunks would lose context; larger chunks would dilute relevance in retrieval. The 100-character overlap prevents information loss at chunk boundaries.
  • Top-K Retrieval (k=3-5): Retrieving 3-5 chunks balances context richness with prompt length constraints. Too few chunks risk missing relevant information; too many add noise and exceed token limits. This range was validated through empirical testing.
  • Future Enhancements: Implementing re-ranking for improved retrieval precision, adding conversational memory for multi-turn dialogues, integrating feedback loops for continuous improvement, and expanding to multilingual support for diverse patient populations.
